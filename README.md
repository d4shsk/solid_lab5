## Выбранный принцип: Dependency Inversion Principle (DIP) — Принцип инверсии зависимостей

### Формулировка принципа:
1. Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба класса должны зависеть от абстракций.
2. Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.

### Предметная область: Уведомление о заказах в интернет-магазине

Когда клиент оформляет заказ, система должна отправить ему уведомление (например, по электронной почте, SMS или в Telegram).

### 1. Нарушение принципа (Код с ошибкой)

В файле violation.py класс верхнего уровня OrderProcessor напрямую зависит от класса нижнего уровня EmailSender:

```python
class OrderProcessor:
    def __init__(self) -> None:
        # Жесткая зависимость. OrderProcessor сам создает конкретный экземпляр EmailSender.
        self._email_sender = EmailSender()
```

#### Проблемы данного подхода (Нарушения):
1. **Жесткая связь:** Мы не можем протестировать `OrderProcessor` изолированно (например, подменив `EmailSender` заглушкой/моком), чтобы не отправлять реальные письма во время тестов.
2. **Нарушение OCP (Open/Closed Principle):** Если бизнес потребует добавить отправку уведомлений по SMS или Telegram, нам придется модифицировать код самого `OrderProcessor`, что повышает риск внесения ошибок в существующий код обработки заказов.

### 2. Исправление нарушения (Рефакторинг)

В файле correct.py мы провели рефакторинг:

1. **Создали абстракцию:** Абстрактный базовый класс NotificationSender с использованием модуля `abc`.
2. **Направили зависимости к абстракции:**
   - Класс верхнего уровня OrderProcessor теперь зависит только от абстрактного класса NotificationSender.
   - Различные реализации отправки (EmailNotificationSender, SmsNotificationSender, TelegramNotificationSender) наследуются от NotificationSender и реализуют его метод `send`.
3. **Внедрение зависимостей (Dependency Injection):** Зависимость передается в конструктор OrderProcessor извне.

```python
from abc import ABC, abstractmethod

class NotificationSender(ABC):
    @abstractmethod
    def send(self, recipient: str, message: str) -> None:
        pass

class OrderProcessor:
    # Теперь зависимость передается извне через конструктор (Constructor Injection).
    def __init__(self, notification_sender: NotificationSender) -> None:
        self._notification_sender = notification_sender
```

#### Преимущества исправления:
- **Слабая связанность:** `OrderProcessor` больше не знает о конкретных механизмах отправки сообщений (SMTP, SMS-шлюзы, Telegram API).
- **Масштабируемость:** Добавление нового канала уведомлений (например, WhatsApp или push-уведомления) требует только создания нового класса, реализующего `NotificationSender`, без изменения существующей логики в `OrderProcessor`.
- **Тестируемость:** Мы можем легко передать мок-объект в конструктор `OrderProcessor` для юнит-тестирования.
